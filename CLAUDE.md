## Project overview

`my-react-app` is a single-page React application (Create React App / `react-scripts` 5)
that is deployed as a **static site** to AWS. The compiled assets are hosted in a
**private S3 bucket** and served globally through a **CloudFront** distribution.

## Architecture

```
                 ┌──────────────┐      OAC (signed)      ┌──────────────────┐
  Users ───────▶ │  CloudFront  │ ─────────────────────▶ │  S3 (private)    │
  (HTTPS)        │ distribution │   origin access ctrl    │  static assets   │
                 └──────────────┘                         └──────────────────┘
```

- **React app** — source in `src/`, static template/assets in `public/`.
- **S3 bucket** — origin of truth for built assets. Private; **no public access**.
  All public reads go through CloudFront via **Origin Access Control (OAC)**.
- **CloudFront** — TLS termination, edge caching, SPA routing (404/403 → `/index.html`).
- **Terraform** (`terraform/`) — declares the S3 bucket, OAC, CloudFront distribution,
  and bucket policy. State is local by default (`terraform/terraform.tfstate`).

> A `Dockerfile` (nginx) also exists for container-based hosting, but the primary
> production path is **S3 + CloudFront** described above.

## Build step

The site is built with Create React App:

```bash
npm ci          # clean, reproducible install from package-lock.json
npm run build   # outputs the production bundle to ./build/
```

**Important:** the deployable artifact is the **`build/` directory**, *not* the
project root. Any S3 sync must target `build/` (e.g. `aws s3 sync build/ s3://<bucket>`).
`build/` is git-ignored and produced fresh in CI.

## Deployment workflow

1. **Provision infra** (one-time / on change) with Terraform:
   - `/scaffold-terraform` — generate/refresh the Terraform config.
   - `/tf-plan` — preview changes.
   - `/tf-apply` — create/update S3 + CloudFront.
2. **Ship the app**:
   - Local: `/deploy` — `npm run build` → `aws s3 sync build/` → CloudFront invalidation.
   - CI: push to `main` triggers the GitHub Actions workflow generated by
     `/setup-gh-actions create`, which runs `npm ci && npm run build`, syncs `build/`
     to S3, and invalidates the CloudFront cache.
3. **Audit** with `/infra-audit` (security + cost + drift).

## Key commands

| Command          | Purpose                                            |
| ---------------- | -------------------------------------------------- |
| `npm start`      | Local dev server (http://localhost:3000)           |
| `npm run build`  | Production build → `build/`                         |
| `npm test`       | Run tests in `src/tests`                           |
| `terraform -chdir=terraform plan`  | Preview infra changes              |
| `terraform -chdir=terraform apply` | Apply infra changes                |

## Conventions & guardrails

- Region: **us-east-1** (CloudFront ACM certs must live here).
- The S3 bucket is **private**; never enable public ACLs or website hosting on it.
- Never commit `terraform.tfstate`, `*.tfvars` with secrets, or AWS credentials.
- Destructive actions (`terraform destroy`, `aws s3 rb`, `rm -rf`) are gated by the
  `pre-tool-guard` hook — confirm intent before overriding.
